<MachineNote>

I wrote these lessons. The source is Nico's session history — roughly 500 prompts and the decisions behind a few months of shipped apps. I read it back, found the habits that recurred, and wrote them up. Prompts marked **[real]** are his exact words; the **[crafted]** ones are cleaned-up models of the same move. Nico chose which lessons made the cut. See [tenets](/tenets) for why this is marked.

</MachineNote>

Fourteen habits, each with an example from the work and a prompt you can copy.

## 1. Plan before you code — make it explain the plan back

Get the approach and open questions on the table before any non-trivial feature. prntd went spec → stack → scaffold → build. → [github.com/nicolovejoy/prntd](https://github.com/nicolovejoy/prntd)

> **[real]** "Let's plan this out and then clear and build. But ask more questions and think about ways to improve our code quality as we go."

## 2. Scope tight — name what not to build

prntd's MVP excluded image upload, marketplace, multi-item orders, and saved designs — and shipped in three days. → [github.com/nicolovejoy/prntd](https://github.com/nicolovejoy/prntd)

> **[crafted]** "Here's the v1 scope. Explicitly out of scope: [list]. If you think something out-of-scope is necessary, flag it — don't just build it."

## 3. Verify end-to-end yourself

Passing tests aren't proof. On ibuild4you, clicking through caught a `github_repo` misconfiguration no test covered. → [github.com/nicolovejoy/ibuild4you](https://github.com/nicolovejoy/ibuild4you)

> **[crafted]** "Walk me through exactly how I verify this works in production, step by step. I'll click through it myself and report back."

## 4. Push back — ask it to think again

Declined a one-consumer abstraction on ibuild4you; rejected bundling visits/costs/feedback into one shared library in prompt-lab. → [github.com/nicolovejoy/ibuild4you](https://github.com/nicolovejoy/ibuild4you) · [github.com/nicolovejoy/prompt-lab](https://github.com/nicolovejoy/prompt-lab)

> **[real]** "This seems like the right way to go, but think through it one more time to be sure."

## 5. Match the codebase, don't impose a style

On ibuild4you, chose plain TypeScript and lockstep tests over a Zod single-source-of-truth to match the project's existing style. → [github.com/nicolovejoy/ibuild4you](https://github.com/nicolovejoy/ibuild4you)

> **[crafted]** "Match the conventions already in this file/folder. Don't introduce new patterns or libraries unless you flag it and explain why first."

## 6. Keep changes reversible

prompt-lab renames canonicalize-on-read — a reversible INSERT, not a data mutation. A defunct email was edited out of `.gitconfig` rather than deleting the file. → [github.com/nicolovejoy/prompt-lab](https://github.com/nicolovejoy/prompt-lab)

> **[real]** "That's an old work account that's defunct, it's in my `.gitconfig`. Should I just delete that file?"

## 7. Write decisions where the AI reads them next time

Plans live in-repo (`docs/`, `CLAUDE.md`), not scratch files. prompt-lab publishes conventions once to a cross-project bulletin board. → [github.com/nicolovejoy/prompt-lab](https://github.com/nicolovejoy/prompt-lab)

> **[real]** "It would be nice to have a bulletin board that this project owns and publishes to, that other projects can subscribe to."

## 8. Name who's who; write specs implementation-blind

A roles note kept the builder and broker straight across sessions (byside). ibuild4you's maker spec was written without showing the current UI, so agents wouldn't anchor on it.

> **[crafted]** "I'm the builder; [name] is the client. Describe the ideal flow from scratch — don't look at the current implementation, I don't want you anchored on it."

## 9. Pivot when a tool isn't working

prntd switched from Flux to Ideogram v3 Turbo when text-on-shirt rendering — the whole product — kept failing. → [github.com/nicolovejoy/prntd](https://github.com/nicolovejoy/prntd)

> **[real]** "Maybe the model isn't the best after all... what would other, better options be — or a combo of two, one for the image, one for the words?"

## 10. Handle credentials carefully from day one

ibuild4you used a dedicated S3-only IAM user and per-service tokens. After a billing scare, one workspace's keys rotated without touching anything else. → [github.com/nicolovejoy/ibuild4you](https://github.com/nicolovejoy/ibuild4you)

> **[real]** "I've created a new workspace and new API keys. Can we test that the new setup works?"

## 11. Defer with a written upgrade path

ibuild4you gated a widget with a magic URL and documented the passcode-auth upgrade trigger. Rate-limiting was deferred the same way. → [github.com/nicolovejoy/ibuild4you](https://github.com/nicolovejoy/ibuild4you)

> **[crafted]** "Let's not build [X] yet. Write a one-line note: what we did instead, and the signal that should make us revisit it."

## 12. Keep the instruction file lean

Trimmed a `CLAUDE.md` from 304 to 101 lines (musicforge); questioned the `/readup` command for doing too much on every launch. → [github.com/nicolovejoy/prompt-lab](https://github.com/nicolovejoy/prompt-lab)

> **[real]** "What is the heavy part of the current readup command? Can we make it less heavy because it's unnecessary?"

## 13. Debug by instrumenting, not guessing

A transient "Invalid token" error on ibuild4you got diagnostic logging on `verifyIdToken`, so the next incident shows the real cause. → [github.com/nicolovejoy/ibuild4you](https://github.com/nicolovejoy/ibuild4you)

> **[crafted]** "Don't guess at the fix. First add logging that tells us exactly what's happening, then reproduce and read the real error before changing anything."

## 14. Delegate the plumbing, own the decisions

prntd wired six external services (Turso, Better-Auth, Stripe, Printful, Replicate, Cloudflare R2); ibuild4you stood up the full S3 stack. Nico made the stack and scope calls. → [github.com/nicolovejoy/prntd](https://github.com/nicolovejoy/prntd) · [github.com/nicolovejoy/ibuild4you](https://github.com/nicolovejoy/ibuild4you)

> **[crafted]** "You handle the wiring — CI, env setup, integration boilerplate. Surface every real decision to me instead of picking. I'll decide; you build."

## The one-line version

Plan first. Scope tight. Verify yourself. Push back. Keep it reversible. Write it down. Delegate the plumbing, own the judgment.
